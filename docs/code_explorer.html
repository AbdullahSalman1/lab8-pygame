⚠️ Potential Issues (2)

✨ Magic Numbers Everywhere

Constants like 120 (flee_radius), 25 (SQUARE_COUNT), and color ranges (80–255) are hardcoded throughout.

Basics: Magic numbers make code harder to read. Why 120? Why 80–255? Future developers must hunt through code to understand intent.
Engineering: If you want to tweak the flee radius from 120 to 150, you must find and change it wherever it appears—error-prone.
Architecture: Violates the DRY (Don't Repeat Yourself) principle and the single source of truth pattern.
🔧 Extract all magic numbers to named constants at the top: FLEE_RADIUS = 120, COLOR_MIN = 80.
🔗 O(n²) Flee Algorithm

The nested loop checks every small square against every big square each frame—quadratic complexity.

Basics: With 25 squares, that's 625 comparisons per frame. At 100 FPS, you're doing 62,500 distance calculations/sec.
Engineering: Fine for a small demo, but scales terribly. Double the squares: 2,500x comparisons instead of 625x—you'd notice lag.
Architecture: A production game would use spatial partitioning (quadtrees, grid cells) so each check is ~O(1) instead of O(n).
🔧 For now, add a # TODO comment noting this. Later, learn spatial indexing to scale to thousands of objects.

Performance Analysis

Score: 2/6 | Primary concern: quadratic flee-checking algorithm. Otherwise, frame rate is stable.

🚀 Performance Wins
Efficient Position Updates
Each square's position is updated in O(1) time. No nested loops or expensive operations.
Framerate Independence
Using clock.tick(FPS) ensures simulation runs consistently at 100 FPS regardless of rendering speed.
🐢 Performance Risks
Quadratic Flee Algorithm
Nested loop checks 25 × 25 = 625 distance calculations per frame. At 100 FPS, that's 62,500 calcs/sec—noticeable with hundreds of squares.
Distance Recomputation
calculate_distance() is called twice per flee check (in should_flee() and again in flee_away())—redundant math.
Verdict: Current performance is acceptable for 25 squares at 60–100 FPS. However, the O(n²) algorithm is the scalability bottleneck. For production, implement spatial partitioning (grid or quadtree) to achieve O(n log n) or O(1) per-object behavior.

Use TypedDict for Square Structure
Define a Square TypedDict with keys: x, y, vx, vy, size, color. This gives IDE autocomplete and type-checker validation. Enables: def update_square(sq: Square) → None.

Spatial Partitioning for Flee Logic
Instead of checking all pairs, partition the screen into a grid. Only check squares in adjacent cells. Reduces flee checks from O(n²) to O(n). Research: quadtrees, KD-trees, spatial hashing.
3
Pygame Sprite Class
Instead of raw dicts, inherit from pygame.sprite.Sprite. Gives you built-in collision detection (pygame.sprite.spritecollide()) and group management.
4
Extract Magic Numbers to Constants
Create FLEE_RADIUS = 120, COLOR_BRIGHTNESS_MIN = 80, etc. at module level. Makes the code's intent clearer and easier to tune.
5
Unit Testing Your Functions
Write tests for calculate_distance(), should_flee(), and flee_away(). Verify edge cases: distance = 0, negative coordinates, boundary squares.
